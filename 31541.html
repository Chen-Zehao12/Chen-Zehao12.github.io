  • 我们需要在哪里证明我们是我们？

1.2 用户名 - 密码

  • 什么是用户名和密码？
  • 它是怎么工作的？

1.2.1 验证 -1

  • 检查用户名和密码
  • 如果有任何一个失败就报告

1.2.2 验证 -2

  • 先检查用户名后检查密码
  • 不匹配的话就报告

1.2.3 验证 -3

  • 检查表中是否存在用户名，然后比较密码
  • 不匹配的话就报告

1.2.4 原理

  • 用户名：标识符
  • 密码：共享密钥
  • 直接比较密码，看他们是否相同的

1.3 CRAM

基于共享密钥的质询 - 响应认证

  • 假设
    • Alice 需要向 Bob 证明自己。Alice 和 Bob 有一个共同的密钥 s
  • 步骤
    • Bob 向 Alice 发送了一个质询 c
    • Alice 使用共享密钥 s 加密 c，生成 e
    • Bob 解密 e 并与 c 进行比较

1.3.1 基于共享密钥的 CRAM

  • 选择性能
    • 我们能避免加密 / 解密吗
  • 过程
    • Bob 向 Alice 发送了一个质询ccc
    • Alice 生成h=hash(c,s)h=hash(c,s)h=hash(c,s)
    • Bob 生成 h′=hash(c,s)h'=hash(c,s)h′=hash(c,s) 并且比较 h′h'h′ 和hhh

1.3.2 基于共享密钥的质询 - 响应的双向认证

  • Bob 向 Alice 发送了一个质询RBR_BRB​
  • Alice 用 sss 加密 RBR_BRB​ 生成了KAB(RB)K_{AB}(R_B)KAB​(RB​)
  • Alice 向 Bob 发送了 KAB(RB)K_{AB}(R_B)KAB​(RB​) 和一个新的质询RAR_ARA​
  • Bob 解密 KAB(RB)K_{AB}(R_B)KAB​(RB​) 来验证 Alice 的身份，并且用 sss 加密 RAR_ARA​ 生成了KAB(RA)K_{AB}(R_A)KAB​(RA​)
  • Bob 向 Alice 发回了KAB(RA)K_{AB}(R_A)KAB​(RA​)
  • Alice 验证KAB(RA)K_{AB}(R_A)KAB​(RA​)

1.4 基于 KDC 的认证（使用密钥分发中心进行认证）

  • 场景介绍
    • KDC—可信的第三方
    • 每个个体都有一个与 KDC 共享密钥
  • Alice 告诉 KDC 她想使用 ksk_sks​ 与 Bob 通信
  • KDC 告诉 Bob 使用 ksk_sks​ 与 Alice 通信
  • Alice 和 Bob 都使用他们的共享密钥向 KDC 进行身份验证

1.4.1 基于 KDC 的认证攻击

  • Trudy 让 Alice 和 Bob 交流
  • Trudy 拦截从 Alice 发向 KDC 和发向 Bob 的所有信息
  • Trudy 重放所有信息

2. 单点登录

2.1 Kerberos

  • 主要有三个角色的存在

    • 访问服务的 Client
    • 提供服务的 Server
    • KDC 密钥分发中心，包括 AS 和 TGS
      • AS: The Authentication Server
      • TGS: The Ticket Granting Server

  • 目标：Alice 登陆一次，可访问多个服务，Bob 代表一个服务

  • Alice 和 AS 有共享KAK_AKA​

  • AS 和 TGS 有共享KTGSK_{TGS}KTGS​

  • TGS 和 Bob 有共享KBK_BKB​

  • AS(Authentication Service): 让 TGS 知道这是 Alice

  • 用 KAK_AKA​ 同时验证 Alice 是否是 Alice

  • KS(t)K_S(t)KS​(t)是给 TGS 来确认是否为 Alice

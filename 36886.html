1.6 基本技术

  • 替换
  • 置换

1.6.1 替换

  • 一个二进制字被另一个二进制字代替
  • 整个替换函数构成键
  • 如果使用 n 位字，那么键空间是2n!2^n!2n!
  • 类似于查找表，有 nnn 个地址行 (因此有2n2n2n 个地址)，每个 nnn 位宽是输出值

1.6.2 置换

  • 二进制字的位被重新排序(排列)
  • 如果我们使用 nnn 位的单词，键空间是n!n!n!(不如替换安全)
  • 这相当于实践中的交叉布线

1.8 混乱与扩散

  • 混乱
    • 一种力求使密文的统计信息和加密密钥的值之间的关系尽可能复杂的技术。加密使用密钥和明文。混乱可以确保密文不提供任何与纯文本相关的线索
  • 扩散
    • 一种通过将每个单独的明文数字的影响分散到多个密文数字上，从而试图模糊明文的统计结构的技术

2.1 ECB(Electronic Codebook)

  • 消息被分解成独立的块，这些块被加密

  • 每个块是一个被替换的值，就像一个代码本，因此命名

  • 每个块都独立于其他块进行编码

    Ci=enc(k1,mi)C_i = enc(k_1, m_i) Ci​=enc(k1​,mi​)

2.1.1 ECB 填充

  • 最后一个字节作为填充指示器
  • PI 表示填充的字节数

2.1.2 ECB 的优缺点

  • 消息重复可能以密文形式显示
    • 如果与消息块对齐，特别是数据，比如图形，或者使用变化很小的消息，这会成为一个代码本分析问题
  • 弱点在于加密的消息块是独立的
  • 主要用途是发送几个数据块

2.2 CBC(Cipher Block Chaining)

  • 这种模式是先将明文切分成若干小段，然后每一小段与初始块或者上一段的密文段进行异或运算后，再与密钥进行加密。
  • 使用初始向量启动进程

Ci=enc(ki,(mi⨂Ci−1))C−1=IVC_i=enc(k_i,(m_i \bigotimes C_{i-1}))\\ C_{-1} = IV Ci​=enc(ki​,(mi​⨂Ci−1​))C−1​=IV

  • 用途：批量数据加密、认证

2.2.1 CBC 优缺点

  • 密文块依赖于它前面的所有块
  • 对一个块的任何更改都会影响后面的所有密文块
  • 需要初始向量
    • 这是发送者和接收者必须知道的内容
    • 如果是明文发送，攻击者可以改变第一个块的位，并改变 IV 来补偿
    • 因此 IV 必须是一个固定值 (如 EFTPOS) 或必须在剩余信息之前以 ECB 模式加密发送

2.3 CFB(Cipher Feedback)

  • 消息被视为比特流添加到分组密码的输出中，结果是下一阶段的反馈(因此得名)
  • 标准允许任何数目的位 (1,8,64 或 128 位等) 被反馈，分别被记作 CFB-1, CFB-8, CFB-64, CFB-128 等等
  • 最有效地使用块中的所有位(64 位或 128 位)

  • 用途：流数据加密、认证

2.3.1 CFB 插入攻击

2.3.2 CFB 优缺点

  • 当数据以位 / 字节形式到达时是合适的
  • 最常见的流加密模式
  • 缺点是做块加密后每个 n 位时需要暂停
  • 请注意，块密码在两端的加密模式中使用，项目错误在错误之后传播几个块

2.4 OFB(Output Feedback)

  • 消息被视为比特流，将密码的输出添加到消息中，然后输出被反馈(因此得名)
  • 反馈是独立于信息的，可以预先计算

Ci=mi⨂OiOi=enc(k1,Oi−1)O−1=IVC_i=m_i \bigotimes O_i\\ O_i=enc(k_1, O_{i-1})\\ O_{-1}=IV Ci​=mi​⨂Oi​Oi​=enc(k1​,Oi−1​)O−1​=IV

  • 用途：噪声信道上的流加密

2.4.1 OFB 优缺点

  • 比特错误不会传播
  • 更容易受到消息流修改的影响
  • Vernam 密码的变体，因此绝不能重用相同的序列(密钥 +IV)
  • 发送方和接收方必须保持同步
  • 最初用 m 位反馈指定
  • 随后的研究表明，只有完整的区块反馈 (即 CFB-64 或 CFB-128) 才应该被使用

2.5 CTR(Counter)

  • 类似于 OFB，但加密的是计数器值，而不是任何反馈值

  • 必须有一个不同的键和计数器值为每个明文块(从来没有重用)

    Ci=mi⨂OiOi=enc(k1,i)C_i=m_i \bigotimes O_i\\ O_i=enc(k_1,i) Ci​=mi​⨂Oi​Oi​=enc(k1​,i)

  • 计算器模式不常见，在 CTR 模式中， 有一个自增的算子，这个算子用密钥加密之后的输出和明文异或的结果得到密文，相当于一次一密。这种加密方式简单快速，安全可靠，而且可以并行加密，但是在计算器不能维持很长的情况下，密钥只能使用一次。

  • 用途：高速网络加密

2.5.1 CTR 优缺点

  • 效率
    • 可以并行加密 h/w 或 s/w
    • 可以提前进行预处理
    • 对于突发的高速连接来说很好
  • 随机访问加密的数据块
  • 可证明的安全性(与其他模式一样好)
  • 但必须确保永远不会重用键 / 计数器值，否则可能会破坏(cf OFB)

3. 非对称密钥加密

3.1 私钥加密密码学

  • 单钥
    • 加密

  • 缺点

3.2 公钥加密密码学

3.2.1 疑问

  • 我们能区分发送者和接收者吗?
  • 我们可以使用不同的密钥加密和解密吗?
  • 我们可以节省密钥分发的精力吗?

3.2.2 基本概念

  • 密钥对
    • 公钥
    • 私钥
  • 公钥加密
  • 私钥解密
  • 使用公钥加密的数据只能使用私钥解密
  • 区分发送者和接收者

3.2.3 主要目的

解决：

  • 密钥分配
  • 数字签名

3.2.4 特点

  • 给定加密密钥，无法计算出解密密钥
  • 使用给定的加密 / 解密密钥易于加密 / 解密
  • 任何一个密钥都可以用于加密，而另一个只能用于解密

3.2.5 应用

  • 加密解密
  • 数字签名
  • 密钥交换

3.2.6 公钥方案的安全性

  • 暴力破解
    • 理论可行
    • 实际很难
  • 基于难问题
  • 让困难的问题变得难以解决
  • 需要非常大的数字才能打破它

3.6 数字签名

3.7 消息验证

  • 消息验证涉及到：
    • 保护消息的完整性
    • 验证发起人的身份
    • 不否认起源(争议解决)
  • 信息签名的电子等价物
  • 一个验证器、签名或消息验证码 (MAC) 随消息一起发送
  • MAC 是通过一些算法生成的，它依赖于消息和一些只有发送方和接收方知道的 (公共或私有) 密钥
  • 消息可以是任意长度
  • MAC 可以是任意长度，但通常是固定大小，如果身份验证方案无法实现，则需要使用某种哈希函数将消息压缩到所需的大小
  • 需要考虑消息和 MAC 的重放问题
    • 需要消息序列号、时间戳或协商的随机值

3.7.2 使用私钥密码的身份验证

  • 如果使用只有发送方和接收方知道的会话密钥对消息进行加密，则还可以对消息进行身份验证

    • 因为只有发送方或接收方可以创建它

    • 任何干扰都将破坏消息(只要它包含足够的冗余来检测变化)。

    • 这并不提供不可抵赖性，因为无法证明是谁创建了消息

3.7.3 信息认证码 MAC

  • MAC 提供身份验证

  • 消息可以加密以保密

    • 通常使用单独的密钥
    • 可以在加密前或加密后计算 MAC
    • 一般认为加密前做得更好

  • 为什么要用 MAC？

    • 有时只需要验证
    • 有时需要验证比加密持久(例如，存档使用)

  • 请注意，MAC 不是数字签名

  • 一个 MAC 是一个密码校验和MAC=checksum(k,m)MAC = checksum(k, m)MAC=checksum(k,m)

    • checksum 是一个函数
    • 压缩变长消息 m
    • 使用一个密钥kkk
    • 固定大小的输出

  • N 对 1 函数 / 匹配

    • 潜在碰撞
    • 但很难找到碰撞

  • 需求

    • 已知MAC=checksum(k,m)MAC = checksum(k, m)MAC=checksum(k,m)，不可能找到MAC=checksum(k,m′)MAC = checksum(k, m')MAC=checksum(k,m′)
    • MAC 均匀分布
    • MAC 同样依赖于消息的所有比特
  • 基于哈希的 MAC(HMAC)
  • HMAC 伪代码

1. 访问控制结构

1.1 访问控制矩阵(ACM)

  • 权限判定复杂度为O(1)O(1)O(1)
  • S: 主体
  • O: 客体
  • A: 访问操作

  • 访问控制矩阵是

    • 一个抽象的概念

    • 不太适合直接执行

      • 如果有成千上万个文件和数百个用户(导致数百万个矩阵条目)，那么管理矩阵可能会非常困难。
      • 矩阵很可能是非常稀疏的，因此实现是低效的
  • 访问控制矩阵通常是稀疏的，可以按行或列分解
  • 两种实现，取决于你如何分解：
    • 访问控制表
    • 访问能力表

1.3 访问能力表

  • 专注于主体
    • 访问权限存储在主体中
    • 访问控制矩阵的权限行

  • 性能和分布式系统安全性之间的良好匹配

    • 安全策略必须处理漫游问题

  • 问题的权限

    • 如何检查谁可能访问一个特定的对象？

    • 如何撤销一个权限？

  • 权限的保护和真实性
    • 如果在单个系统中使用
      • 可以依赖于操作系统的完整性和它所使用的机制
    • 如果通过网络使用
      • 真实性和保护主要是加密的

1.4 访问控制表(ACLs)

  • 不需要写没有的权限
  • 从 NULL 开始，单调递增
  • 权限判定复杂度=[OJ]+[S]=[OJ] +[S]=[OJ]+[S]
  • 专注于客体
    • 访问权限与对象一起存储
    • 访问控制矩阵的 ACL 列

  • 访问权限通常是为用户组定义的

    • 因为单个主体可能会形成一个巨大的列表

  • ACLs 是操作系统安全性的典型代表

  • 在 UNIX 中，ACLs 被附加到文件中

1.5 聚合技术

  • ACLs 和访问能力表的用途有限(一个关注主体，另一个关注客体)
  • 需要聚合主体和客体
    • 组
    • 角色
    • 程序
    • 数据类型

1.6 组和负面权限

  • 组是用户和客体之间的中间层
  • 为处理特殊情况，负面授予撤回权限

1.7 基于角色的访问控制(RBAC)

  • 在主体和客体之间可以插入几个中间概念

  • Data types: 数据类型是具有相同结构的一组对象(如银行账户)。

    • 每个对象都具有特定的数据类型，只能通过为该数据类型定义的过程来访问。

  • Procedures: 具有比读或写更复杂语义的高级访问控制方法

    • procedures 只能应用于某些数据类型的对象; 例如: 银行账户之间的资金转账。

  • Roles: 分配给 roles 的 procedures 的集合；一个用户可以有多个角色，一个角色可以有多个用户

  • 例如

    • Objects 是银行账户

    • Subjects 是银行员工

    • 银行账户的集合形成了一个数据类型

    • 我们定义角色

      • Teller
      • Clerk
      • Administrator

    • 我们定义过程

      • 发放账户(CA)

      • 代理账户(DA)

      • 在账户之间转移资金(TF)

      • 创建账号(NA)

    • 我们分配过程
      • Teller 能够做 CA 和 DA
      • Clerk 能够做 TF
      • Administrator 能够做 NA
      • Administrator 能够做所有的过程
  • 角色很适合业务中的典型访问控制需求
  • 角色实现于
    • Window NT onwards (as global and local groups)
    • IBM’s OS/400
    • Oracle 8 onwards
    • .NET framework
  • RBAC 没有普遍接受的标准
  • 美国国家标准 RBAC
  • 银行中的 RBAC

1.8 Lattices

  • 假设一个主体只有在主体的标号大于或等于客体的标号时才能观察到客体。
  • 格是一种数学结构，其中这些问题有独特的答案
    • 给定两个具有不同标签的客体，一个主体必须被允许观察两个客体的最小标签是什么?
    • 给定两个有不同标签的主体，一个客体能被两个主体观察到的最大标签是什么?
  • 格是一个部分有序集，其中每一对元素都有一个最大下界和一个最小上界

  • System Low and System High

    • 如果 a≤b，我们说“a 被 b 支配”或“b 支配 a”
    • 如果一个标签被所有其他标签所支配，那么它将被称为 System Low
    • 如果一个标号优于所有其他标号，则称为 System High
    • 什么是 System Low 和 System High 在幂集格的例子？

  • A Flat Lattice

2. 访问控制模型和策略

  • 安全策略

    • 获取企业的安全性需求，或者描述为了实现安全性而必须采取的步骤

  • 安全模型

    • 是安全策略的正式描述

  • 例子：信息流策略

    • 解决保密要求
    • 我们假设存在一个安全标签格
    • 使用安全函数为每个主体和客体分配一个安全标签
    • 当 (x)≤(y) 时，信息可以从实体 x 流向实体 y
      • 信息可以从低安全性实体流向高安全性实体
    • 根据信息流原理定义读写访问权限

  • 读权限

    • 信息从一个客体 o 流向一个主体 s
    • 当 (o)≤(s) 时，授予读访问权限
    • 如果您的安全标签大于客体的安全标签，则可以读取该客体
    • 这个条件在 BLP 术语中称为“无读取”或简单安全性 (ss) 属性

  • 写权限

    • 信息从主体 s 流向客体 o
    • 当 (s)≤(o) 时，允许写访问
      • 如果你的安全标签比客体的安全标签小，你可以写入一个客体
      • 相当反直觉，但有必要防止违反保密等
        • 一个高度机密的用户正在向一个不安全的打印机写信
    • 这个条件在 BLP 术语中称为“无记录”或∗−属性(星形属性)
    • 没有读取和写属性是 BLP 的“强制访问控制”属性

  • Information Flow Blocked by ∗ − Property

2.1 保密政策

  • 概述
    • 目标：防止未经授权的信息泄露
      • 处理信息流
      • Integrity incidental
    • 多级安全模型是最著名的例子
      • Bell-LaPadula Model

2.2 Bell-LaPadula 模型，第一步

  • 在机密性级别上：不读高，不写低
  • 安全级别按线性顺序排列
    • 绝密：最高
    • 机密
    • 保密
    • 非保密：最低
  • 级别包括安全许可 L(s)
    • 客体的安全级别为 L(o)
  • 例如
  • Tamara 可以读取所有文件
  • Claire 无法阅读人事或电子邮件文件
  • Ursula 只能阅读电话名单

  • Reading Information

    • 信息向上流动，而不是向下流动
      • “读上”不允许，“读下”允许
    • 简单安全条件(步骤 1)
      • 主体 s 可以在 L(o)≤L(s)的情况下读取对象 o，且主体 s 具有读取 o 的权限
        • 注: 结合强制控制 (安全级别关系) 和酌情控制(所需许可)
      • 有时被称为“不读上”规则

  • 基本安全定理，第一步

    • 如果系统最初处于安全状态，并且系统的每个转换都满足简单的安全条件，即步骤 1 和 *- 属性，即步骤 1，那么系统的每个状态都是安全的
    • 证明: 归纳转换的数量

2.3 Bell-LaPadula 模型，第二步

  • 扩展安全级别的概念以包括类别
  • 安全级别为(许可，类别设置)
  • 例如
    • (Top Secret, NUC, EUR, US)
    • (Confidential, EUR, US)
    • (Secret, NUC, US)

  • 等级和格

    • (A, C)dom(A′, C′) iff A′ ≤ A and C′ ⊆ C Examples
      • (Top Secret, NUC, US) dom (Secret, NUC)
      • (Secret, NUC, EUR) dom (Confidential,NUC, EUR)
      • (Top Secret, NUC) ¬dom (Confidential, EUR)
    • Let C be set of classifications, K set of categories. Set of security levels L = C × K, dom form lattice
      • lub(L) = (max(A), C)
      • glb(L) = (min(A), phi)

  • 等级和排序

    • Security levels partially ordered
      • Any pair of security levels may (or may not) be related by dom
    • “dominates”serves the role of “greater than”in step 1
      • “greater than”is a total ordering, though

  • 读信息

    • Information flows up, not down
      • “Reads up”disallowed, “reads down”allowed
    • Simple Security Condition (Step 2)
      • Subject s can read object o iff L(s) dom L(o) and s has permission to read o
        • Note: combines mandatory control (relationship of security levels) and discretionary control (the required permission)
      • Sometimes called “no reads up”rule

  • 写信息

    • Information flows up, not down
      • “Writes up”allowed, “writes down”disallowed
    • *-Property (Step 2)
      • Subject s can write object o iff L(o) dom L(s) and s has permission to write o
        • Note: combines mandatory control (relationship of security levels) and discretionary control (the required permission)
      • Sometimes called “no writes down”rule

  • 基本安全定理，第 2 步

    • 如果系统最初处于安全状态，并且系统的每一次转换都满足简单安全条件 (步骤 2) 和 *- 属性(步骤 2)，那么系统的每一种状态都是安全的
      • 证明: 归纳转换的数量
      • 在实际的基本安全定理中，自由访问控制被视为第三个属性，简单安全属性和 *- 属性的表述消除了定义中的自由访问部分，但这里的表达方式更简单。

2.4 Biba 完整性模型

  • Biba 模型是在 bell-lapadula 模型之后开发的，它跟 bell-lapadula 模型很相似，被用于解决应用程序数据的完整性问题。Bell-lapadula 使用安全级别（top secret,secret,sensitive, 等），这些安全级别用于保证敏感信息只被授权的个体所访问。Biba 模型不关心信息保密性的安全级别，因此它的访问控制不是建立在安全级别上，而是建立在完整性级别上。

  • Biba 模型能够防止数据从低完整性级别流向高完整性级别

  • 跟 Bell-lapadula 一样，Biba 模型也有三条规则提供这种保护

    • 星完整性规则（*-integrity axiom）

      • 完整性级别低的主体不能对完整性级别高的客体写数据

    • 简单完整性规则（simple integrity axiom）

      • 完整性级别高的主体不能从完整性级别低的客体读取数据

    • 恳求属性规则（invocation property）

      • 完整性级别低的主体不能从级别高的客体调用程序或服务

  • 完整性规则上：不写高，不读低，不调高

  • Set of subjects S, objects O, integrity levels I, relation ≤⊆ I × I holding when second dominates first

  • min : I × I → I returns lesser of integrity levels

  • i : S ∪ O → I gives integrity level of entity

  • r:S×Omeanss∈Scanreado∈O

  • w, x defined similarly

  • Intuition for Integrity Levels

    • 等级越高越机密
      • 程序能正确运行
      • 数据是准确和 / 或可靠的
    • Note relationship between integrity and trustworthiness
    • 重点：完整性级别不是安全级别

  • 完整性政策：Biba’s Model

    • 类似于 Bell-LaPadula 模型
      • s∈Ss∈Ss∈S can read o∈Oo∈Oo∈O iff i(s)≤i(o)i(s)≤i(o)i(s)≤i(o)
      • s∈Ss∈Ss∈S can write to o∈Oo∈Oo∈O iff i(o)≤i(s)i(o)≤i(s)i(o)≤i(s)
      • s1∈Ss_1 ∈Ss1​∈S can execute s2∈Ss_2 ∈Ss2​∈S iff i(s2)≤i(s1)i(s_2)≤i(s_1)i(s2​)≤i(s1​)
    • 信息流结果保持不变
    • 虽然是不同的证据

  • 严格完整性策略

    • 简单完整性: I(S)≥I(O)I(S) ≥ I(O)I(S)≥I(O)
    • 完整性约束: I(S)≤I(O)I(S) ≤ I(O)I(S)≤I(O)
    • invocation property: I(S1)≥I(S2)I(S1) ≥ I(S2)I(S1)≥I(S2)

2.5 中国墙模型

  • 问题：

    • 托尼为美国银行提供投资咨询

    • 他被要求为 Toyland Bank 提供投资建议

  • 利益冲突不能接受，因为他给任何一家银行的通知将影响另一家银行

2.5.1 和 Bell-LaPadula 比较

  • 根本不同
    • CW 没有安全标签，B-LP 有
    • CW 有过去访问的概念，B-LP 没有
  • Bell-LaPadula 可以捕获任何时间的状态
    • 每对 (COI, CD) 都获得安全类别
    • Two clearances, S (sanitized) and U (unsanitized) S do m U
    • Subjects assigned clearance for compartments without multiple categories corresponding to CDs in same COI class

  • Bell-LaPadula 不能跟踪随时间的变化

    • 苏珊病了，安娜需要接手

      • C-W 的历史让安娜知道她是否可以
      • No way for Bell-LaPadula to capture this

  • 访问限制随着时间的推移而改变

    • 最初，C-W 中的主体可以阅读任何客体

    • Bell-LaPadula 限制了主体可以访问的客体集

      • Can’t clear all subjects for all categories, because this violates CW-simple security condition
